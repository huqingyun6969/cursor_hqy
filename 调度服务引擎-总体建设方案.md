# 第一节 方案概述

## 1.1 项目背景

视联平台需要一套 **可视化工作流编排与发布引擎**，使相关业务人员能够在 Web 端通过拖拽、连线完成 API 接口的编排，编排完成后一键发布为标准 RESTful API 或是通用微服务(仅限内部调用)，自动注册到微服务中心，通过 **视联平台网关** 和 **能力开放网关** 双入口对外提供服务。类似"能力上架到视联平台"的模式——编排即开发、发布即上架。

## 1.2 系统定位

| 维度 | 说明 |
|------|------|
| 底层引擎 | FlowGram.AI（字节跳动开源的可视化工作流/智能体编排平台） |
| 项目形态 | 智能体全生命周期管理 + 可视化工作流编辑 + 任务运行时 + **流程发布** + **动态路由注册** + 双网关对外服务 |
| 核心价值 | 零代码编排 → 一键发布 → 自动注册 → 网关可调 → 调用可观测 |

**功能架构分层说明**：

| 层次 | 职责 | 核心模块 |
|------|------|---------|
| **用户层** | 不同角色的使用入口 | 编排人员、管理员、订阅方、开发者、运维 |
| **门户与入口层** | 前端页面 + 网关入口 | 管理页、编辑器、能力市场、视联网关、开放网关 |
| **核心功能层** | 六大功能域的具体功能 | 智能体管理、工作流编辑、流程发布、任务执行、能力市场、运行日志 |
| **能力支撑层** | 横切面的通用能力 | 动态路由、鉴权安全、Schema 管理、缓存、限流熔断 |
| **基础设施层** | 底层技术组件 + 信创适配 | FlowGram.AI、MySQL（信创：达梦/金仓）、Redis（信创：东方通TongRDS）、Nacos、Nginx/网关 |

## 2.1 功能域划分

| 功能域 | 说明 | 主要入口 |
|--------|------|----------|
| **智能体管理** | 智能体的 CRUD、搜索、复制、基本信息编辑 | `/manage` 智能体管理页 |
| **工作流编辑** | 画布编辑、节点/连线、表单配置、变量、保存 | `/editor/:id` 编辑器页 |
| **任务执行与运行时** | 启动任务、轮询状态/结果、取消、校验 | 编辑器内「运行」、API |
| **流程发布与上架** | 参数提取、API 定义、动态路由注册、能力上架 | 编辑器「发布」、管理页「发布管理」 |
| **运行日志与执行记录** | 按智能体查看执行记录、状态筛选、详情 | 管理页「运行日志」弹窗 |
| **能力市场与订阅** | 已发布能力浏览、订阅、在线调试 | `/marketplace` 能力市场页 |
| **多端调用** | 前端代理、服务端运行时、Java HTTP、网关调用 | 配置与客户端 SDK |

## 2.2 用户角色与使用场景

| 角色 | 典型场景 |
|------|----------|
| **业务编排人员** | 在管理页创建智能体，进入编辑器设计工作流，保存后点击发布，填写 API 元信息 |

| **运维人员** | 查看运行日志、监控调用量、处理异常告警 |

## 3.3 Spring Boot 动态路由设计

### 3.3.1 设计目标

流程发布后，无需重启服务即可注册新的 HTTP 端点，外部通过统一路径模式调用。

### 3.3.2 路由模式

```
统一上下文: /smart
路由模式:   /smart/api/flow/{flowCode}
HTTP 方法:  POST（默认）
Content-Type: application/json
```

### 3.3.3 实现方案：统一入口 + 路由分发

采用 **统一 Controller 入口 + 数据库路由表** 的方式，而非运行时动态注册 RequestMapping（避免线程安全和热更新问题）：

```java
@RestController
@RequestMapping("/smart/api/flow")
public class FlowDispatchController {

    @Autowired
    private FlowRouteService flowRouteService;

    @Autowired
    private FlowExecutionService flowExecutionService;

    /**
     * 统一流程调用入口
     * 根据 flowCode 路由到对应的已发布流程
     */
    @PostMapping("/{flowCode}")
    public ResponseEntity<FlowApiResponse> execute(
            @PathVariable String flowCode,
            @RequestBody(required = false) Map<String, Object> body,
            @RequestParam Map<String, String> queryParams,
            HttpServletRequest request) {

        // 1. 查找已发布的流程路由
        FlowRoute route = flowRouteService.getActiveRoute(flowCode);
        if (route == null) {
            return ResponseEntity.status(404)
                .body(FlowApiResponse.fail(404, "流程能力不存在或已下架"));
        }

        // 2. 参数校验
        ValidationResult validation = flowRouteService.validateParams(route, body, queryParams);
        if (!validation.isValid()) {
            return ResponseEntity.badRequest()
                .body(FlowApiResponse.fail(400, validation.getMessage()));
        }

        // 3. 组装 inputs 并执行
        Map<String, Object> inputs = flowRouteService.assembleInputs(route, body, queryParams);
        FlowExecutionResult result = flowExecutionService.executeFlow(route, inputs, request);

        // 4. 返回统一响应
        return ResponseEntity.ok(FlowApiResponse.success(result.getOutputs()));
    }
}
```

### 3.3.4 路由缓存策略

```java
@Service
public class FlowRouteService {

    @Autowired
    private FlowRouteRepository flowRouteRepo;

    @Autowired
    private RedisTemplate<String, FlowRoute> redisTemplate;

    private static final String ROUTE_CACHE_PREFIX = "smart:flow:route:";
    private static final long CACHE_TTL_MINUTES = 30;

    public FlowRoute getActiveRoute(String flowCode) {
        String cacheKey = ROUTE_CACHE_PREFIX + flowCode;
        FlowRoute cached = redisTemplate.opsForValue().get(cacheKey);
        if (cached != null) {
            return cached;
        }
        FlowRoute route = flowRouteRepo.findByFlowCodeAndStatus(flowCode, "ACTIVE");
        if (route != null) {
            redisTemplate.opsForValue().set(cacheKey, route, CACHE_TTL_MINUTES, TimeUnit.MINUTES);
        }
        return route;
    }

    public void invalidateRouteCache(String flowCode) {
        redisTemplate.delete(ROUTE_CACHE_PREFIX + flowCode);
    }
}
```

## 3.4 FlowGram Runtime 对接

### 3.4.1 执行服务封装

```java
@Service
public class FlowExecutionService {

    @Autowired
    private RestTemplate restTemplate;

    @Value("${flowgram.agents-server.base-url}")
    private String agentsServerBaseUrl;

    /**
     * 执行流程并等待结果（同步模式）
     */
    public FlowExecutionResult executeFlow(FlowRoute route, Map<String, Object> inputs,
                                            HttpServletRequest request) {
        // 1. 从数据库/缓存加载 schema
        String schemaJson = loadSchema(route.getPublishId());

        // 2. 构建运行请求
        Map<String, Object> runBody = new HashMap<>();
        runBody.put("schema", schemaJson);
        runBody.put("inputs", inputs);

        HttpHeaders headers = new HttpHeaders();
        headers.setContentType(MediaType.APPLICATION_JSON);
        headers.set("x-call-source", "gateway");
        headers.set("x-agent-id", String.valueOf(route.getAgentId()));
        headers.set("x-flow-code", route.getFlowCode());

        // 3. 启动任务
        HttpEntity<Map<String, Object>> entity = new HttpEntity<>(runBody, headers);
        Map<String, Object> runResult = restTemplate.postForObject(
            agentsServerBaseUrl + "/api/task/run", entity, Map.class);

        String taskId = (String) runResult.get("taskID");

        // 4. 轮询等待结果
        return pollForResult(taskId);
    }

    private FlowExecutionResult pollForResult(String taskId) {
        long startTime = System.currentTimeMillis();
        long maxWaitMs = maxWaitSeconds * 1000L;

        while (System.currentTimeMillis() - startTime < maxWaitMs) {
            Map<String, Object> result = restTemplate.getForObject(
                agentsServerBaseUrl + "/api/task/result?taskID=" + taskId, Map.class);

            if (result != null) {
                String status = (String) result.get("status");
                if ("success".equals(status) || "failed".equals(status) || "cancelled".equals(status)) {
                    return FlowExecutionResult.from(result);
                }
            }
            Thread.sleep(pollIntervalMs);
        }
        throw new FlowExecutionTimeoutException("流程执行超时: taskId=" + taskId);
    }
}
```

### 4.2.2 参数自动提取算法

从 `design_json` 的 start 节点中提取 inputs，自动推断类型：

```java
public class InputParamExtractor {

    /**
     * 从 design_json 提取 start 节点的 inputs 参数定义
     * 
     * 输入示例:
     * inputs: {
     *   alarmTimeStart: "2021-3-2 23:59:59",
     *   pageNum: 1,
     *   alarmTimeEnd: "2026-3-10 23:59:59",
     *   pageSize: 6
     * }
     * 
     * 输出: 参数列表，含 name / type / required / defaultValue / description
     */
    public List<FlowParamDefinition> extractInputParams(String designJson) {
        JsonNode root = objectMapper.readTree(designJson);
        JsonNode startNode = findStartNode(root);
        JsonNode inputs = extractInputsFromStartNode(startNode);

        List<FlowParamDefinition> params = new ArrayList<>();
        Iterator<Map.Entry<String, JsonNode>> fields = inputs.fields();

        while (fields.hasNext()) {
            Map.Entry<String, JsonNode> field = fields.next();
            FlowParamDefinition param = new FlowParamDefinition();
            param.setParamName(field.getKey());
            param.setParamType(inferType(field.getValue()));
            param.setRequired(true);  // 默认必填
            param.setDefaultValue(field.getValue().isNull() ? null : field.getValue().asText());
            param.setDescription("");  // 待用户补充
            params.add(param);
        }
        return params;
    }

    private String inferType(JsonNode value) {
        if (value.isTextual()) return "String";
        if (value.isInt()) return "Integer";
        if (value.isLong()) return "Long";
        if (value.isDouble() || value.isFloat()) return "Double";
        if (value.isBoolean()) return "Boolean";
        if (value.isArray()) return "Array";
        if (value.isObject()) return "Object";
        return "String";  // 默认 String
    }
}
```

## 4.3 动态路由注册

### 4.3.1 路由注册流程

```
发布服务
    │
    ├─ 1. 写入 flow_route 表（flowCode, publishId, agentId, status=ACTIVE）
    │
    ├─ 2. 写入 flow_publish 表（含 schema_json 加密存储）
    │
    ├─ 3. 写入 flow_api_params 表（参数定义）
    │
    ├─ 4. 清除 Redis 路由缓存
    │     └─ DEL smart:flow:route:{flowCode}
    │
    └─ 5. 路由立即生效
          └─ 下次请求 /smart/api/flow/{flowCode} 时
             FlowDispatchController 查到 ACTIVE 路由即可调度
```

### 4.3.2 路由规则与约束

| 规则 | 说明 |
|------|------|
| **flowCode 唯一性** | 同一时间只能有一个 ACTIVE 状态的同名 flowCode |
| **flowCode 命名规范** | 仅允许小写字母、数字、短横线，长度 3~64，如 `warning`、`device-query`、`alarm-list` |
| **路由路径格式** | 固定前缀 `/smart/api/flow/` + flowCode |
| **HTTP 方法** | 默认 POST，统一使用 JSON Body 传参 |
| **版本化路由** | 支持 `/smart/api/flow/v2/warning` 形式的版本路由（可选） |
| **路由状态流转** | DRAFT → ACTIVE → INACTIVE（下架） → ACTIVE（重新上架） |
| **并发安全** | 发布/下架操作需加分布式锁（Redis SETNX），防止并发冲突 |
| **优雅下线** | 下架时等待进行中的请求完成（最长等待 30s），再标记 INACTIVE |

### 4.3.3 路由状态机

```
          publish()             unpublish()           republish()
 DRAFT ───────────→ ACTIVE ──────────────→ INACTIVE ──────────────→ ACTIVE
   │                  │                       │
   │   delete()       │   update()            │   delete()
   ▼                  ▼                       ▼
 DELETED         新版本 ACTIVE              DELETED
                 旧版本 INACTIVE
```

### 4.4.2 通用请求规范

#### 请求头（Headers）

| Header | 必填 | 说明 |
|--------|------|------|
| Content-Type | 是 | 固定 `application/json` |
| X-Scene-Code | 视联网关必填 | 视联平台分配的场景码 |
| X-Access-Key | 开放网关必填 | 能力开放平台分配的 AccessKey |
| X-Signature | 开放网关必填 | 请求签名（HMAC-SHA256） |
| X-Timestamp | 开放网关必填 | 请求时间戳（毫秒），有效期 ±5 分钟 |
| X-Nonce | 开放网关建议 | 随机字符串，防重放 |
| X-Request-Id | 否 | 调用方自定义请求追踪 ID |

#### 请求参数（Query）

通常不使用 Query 参数，所有业务参数统一放 Body。但保留扩展能力：

| Query 参数 | 必填 | 说明 |
|-----------|------|------|
| _version | 否 | 指定调用的流程版本号，默认调用最新 ACTIVE 版本 |
| _async | 否 | `true` 时异步执行，立即返回 taskId，调用方自行轮询结果 |
| _timeout | 否 | 自定义超时时间（秒），覆盖默认值 |

#### 请求体（Body）

Body 结构为扁平 JSON 对象，字段与发布时定义的参数一一对应：

### 4.4.4 鉴权方案详解

#### 方案一：SceneCode 简单身份认证（视联平台网关）

```
  │ ──────────────────────────────→│                              │
  │                                │  校验 SceneCode              │
  │                                │  ├─ 是否存在                 │

  │ ←──────────────────────────────│                              │
```

#### 方案二：AccessKey 签名认证（能力开放网关）

适用于第三方系统调用，安全级别更高：

**签名算法**：

```
1. 构造待签名字符串:
   StringToSign = HTTP-Method + "\n"
                + URI + "\n"
                + Timestamp + "\n"
                + Nonce + "\n"
                + SHA256(RequestBody)

2. 计算签名:
   Signature = Base64(HMAC-SHA256(AccessKeySecret, StringToSign))

3. 请求头:
   X-Access-Key: AK20260303xxxxx
   X-Signature: <Signature>
   X-Timestamp: 1741012345678
   X-Nonce: random-string-uuid
```

**调用示例（Java）**：

```java
public class OpenGatewayClient {

    private String accessKey = "AK20260303xxxxx";
    private String accessSecret = "your-secret-key";
    private String gatewayUrl = "https://open-gateway.example.com";

    public String callFlow(String flowCode, String bodyJson) throws Exception {
        String uri = "/smart/api/flow/" + flowCode;
        String timestamp = String.valueOf(System.currentTimeMillis());
        String nonce = UUID.randomUUID().toString();
        String bodyHash = sha256Hex(bodyJson);

        String stringToSign = "POST\n" + uri + "\n" + timestamp + "\n" + nonce + "\n" + bodyHash;
        String signature = base64(hmacSha256(accessSecret, stringToSign));

        HttpRequest request = HttpRequest.newBuilder()
            .uri(URI.create(gatewayUrl + uri))
            .header("Content-Type", "application/json")
            .header("X-Access-Key", accessKey)
            .header("X-Signature", signature)
            .header("X-Timestamp", timestamp)
            .header("X-Nonce", nonce)
            .POST(HttpRequest.BodyPublishers.ofString(bodyJson))
            .build();

        return httpClient.send(request, HttpResponse.BodyHandlers.ofString()).body();
    }
}
```

**调用示例（cURL）**：

```bash
curl -X POST "https://open-gateway.example.com/smart/api/flow/warning" \
  -H "Content-Type: application/json" \
  -H "X-Access-Key: AK20260303xxxxx" \
  -H "X-Signature: <computed-signature>" \
  -H "X-Timestamp: 1741012345678" \
  -H "X-Nonce: random-uuid" \
  -d '{
    "alarmTimeStart": "2021-3-2 23:59:59",
    "pageNum": 1,
    "alarmTimeEnd": "2026-3-10 23:59:59",
    "pageSize": 6
  }'
```

# 第六节 部署与运维

## 6.1 部署架构总体设计

### 6.1.1 设计原则

| 原则 | 说明 |
|------|------|
| **分级部署** | 根据业务重要程度区分核心集群与一般集群，资源配比差异化 |
| **水平可扩展** | 所有服务实例均为无状态设计，支持按需水平扩缩容 |
| **高可用优先** | 核心集群采用多副本 + 跨可用区部署，消除单点故障 |
| **弹性伸缩** | 结合 K8s HPA / 手动扩容，适配业务并发峰谷变化 |
| **故障隔离** | 核心与一般业务物理隔离（独立 Namespace / Node Pool），避免资源争抢 |

### 6.1.2 集群分级定义

| 维度 | 核心业务集群（P0 高可靠） | 一般业务集群（P1 标准） |
|------|------------------------|----------------------|
| **定位** | 承载生产关键流程（告警处理、设备管控、安全合规等） | 承载内部工具、报表查询、非实时批处理等 |
| **可用性目标** | ≥ 99.95%（年停机 ≤ 4.4h） | ≥ 99.9%（年停机 ≤ 8.8h） |
| **部署模式** | 多副本 + 跨可用区 + 主备 DB + Redis Cluster | 多副本 + 单可用区 + DB 主从 + Redis Sentinel |
| **服务最小副本** | smart-flow-engine ≥ 3，agents-server ≥ 3 | smart-flow-engine ≥ 2，agents-server ≥ 2 |
| **弹性上限** | HPA max=20（按 CPU/QPS 自动扩） | HPA max=10 |
| **数据库** | 独立 MySQL 主从 + 读写分离 | 可共享 DB 实例（独立 Schema） |
| **缓存** | 独立 Redis Cluster（≥ 6 节点） | 共享 Redis Sentinel（3 节点） |
| **网络隔离** | 独立 Namespace + NetworkPolicy | 独立 Namespace |
| **发布策略** | 金丝雀发布 / 蓝绿部署 | 滚动更新 |
| **监控告警** | 1 分钟粒度 + 即时告警 + 值班 On-Call | 5 分钟粒度 + 告警通知 |

## 6.2 水平扩展方案

### 6.2.1 服务无状态设计保障

所有服务实例严格无状态，扩缩容不影响业务连续性：

| 服务 | 无状态保障措施 | 说明 |
|------|-------------|------|
| smart-flow-engine | Session 无状态（Token 鉴权）、路由数据存 DB/Redis | 新实例启动即可承载流量 |
| agents-server | 任务状态存 DB、报告缓存设 TTL 自动过期 | 不依赖本地持久化 |
| 前端 Nginx | 纯静态资源，无会话状态 | CDN + 多实例 |

### 6.2.2 Kubernetes HPA 弹性伸缩策略

```yaml
# smart-flow-engine HPA — 核心集群
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: smart-flow-engine-core-hpa
  namespace: smart-core
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: smart-flow-engine
  minReplicas: 3
  maxReplicas: 20
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 30      # 快速扩容
      policies:
        - type: Pods
          value: 4                         # 每次最多扩 4 个 Pod
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300     # 缓慢缩容，避免抖动
      policies:
        - type: Pods
          value: 2
          periodSeconds: 120
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60          # CPU 超 60% 触发扩容
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 70          # 内存超 70% 触发扩容
    - type: Pods
      pods:
        metric:
          name: http_requests_per_second
        target:
          type: AverageValue
          averageValue: "200"             # 单 Pod QPS 超 200 触发扩容
---
```

# 第七节 信创兼容与适配方案

## 7.1 信创适配总体策略

本系统默认基于 **MySQL 8.0 + Redis 6.x + Tomcat（Spring Boot 内嵌）** 构建。当项目需要部署到信创环境时，涉及关系型数据库、缓存中间件、应用服务器、操作系统及 CPU 架构等多层面的适配工作，均需以 **定开方式** 完成。

### 7.1.1 适配原则

| 原则 | 说明 |
|------|------|
| **默认先行** | 标准版本基于 MySQL + Redis 开发交付，信创适配作为独立定开工作包 |
| **抽象隔离** | 通过 ORM 框架 + 方言配置 + 接口抽象隔离底层差异，业务代码无感知 |
| **最小改动** | 优先通过配置切换（数据源、方言、驱动），仅在不兼容时才改写 SQL/代码 |
| **逐一验证** | 每种信创组件单独适配、独立测试，出具兼容性测试报告 |
| **版本锁定** | 信创组件版本需在定开合同中明确锁定，避免后续版本升级引发回归 |

## 7.3 数据库适配详细方案（关系型数据库）

### 7.3.1 SQL 方言差异矩阵

| 语法点 | MySQL 8.0 | 达梦 DM8 | 人大金仓 KingbaseES | 适配方式 |
|--------|-----------|----------|---------------------|---------|
| **自增主键** | `AUTO_INCREMENT` | `IDENTITY(1,1)` | `SERIAL` / `GENERATED ALWAYS AS IDENTITY` | MyBatis-Plus IdType 配置 |
| **大文本类型** | `LONGTEXT` | `CLOB` / `TEXT` | `TEXT` | DDL 脚本分版本维护 |
| **时间默认值** | `CURRENT_TIMESTAMP` | `SYSDATE` / `CURRENT_TIMESTAMP` | `CURRENT_TIMESTAMP` | 统一用 `CURRENT_TIMESTAMP` |
| **ON UPDATE 时间戳** | `ON UPDATE CURRENT_TIMESTAMP` | 不支持，需触发器 | 不支持，需触发器 | 建触发器或应用层赋值 |
| **分页语法** | `LIMIT offset, size` | `LIMIT size OFFSET offset`（兼容模式）或 `TOP` / `ROWNUM` | `LIMIT size OFFSET offset` | MyBatis-Plus 分页插件自动适配 |
| **字符串拼接** | `CONCAT()` | `\|\|` 或 `CONCAT()` | `\|\|` 或 `CONCAT()` | 统一用 `CONCAT()` |
| **布尔类型** | `TINYINT(1)` | `BIT` / `INT` | `BOOLEAN` | DDL 分版本，ORM 层统一映射 |
| **引擎声明** | `ENGINE=InnoDB` | 不需要 | 不需要 | DDL 条件渲染 |
| **字符集** | `CHARSET=utf8mb4` | `UNICODE` 默认支持 | `UTF8` 默认支持 | DDL 条件渲染 |
| **索引语法** | `INDEX idx_name (col)` | 支持 | 支持 | 基本兼容 |
| **UPSERT** | `ON DUPLICATE KEY UPDATE` | `MERGE INTO` | `ON CONFLICT DO UPDATE` | Mapper XML 分方言编写 |

**达梦 DM8 DDL 示例**（对应 MySQL 的 `agents` 表）：

```sql
-- 达梦 DM8 适配版本
CREATE TABLE agents (
    id              BIGINT IDENTITY(1,1) PRIMARY KEY,
    name            VARCHAR(200)  NOT NULL,
    description     VARCHAR(1000) DEFAULT NULL,
    design_json     CLOB          DEFAULT NULL,
    created_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_agents_name ON agents(name);

-- 达梦不支持 ON UPDATE CURRENT_TIMESTAMP，需创建触发器
CREATE OR REPLACE TRIGGER trg_agents_updated_at
BEFORE UPDATE ON agents
FOR EACH ROW
BEGIN
    :NEW.updated_at := CURRENT_TIMESTAMP;
END;
```

**人大金仓 KingbaseES DDL 示例**：

```sql
-- 人大金仓 KingbaseES 适配版本 (PostgreSQL 兼容模式)
CREATE TABLE agents (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(200)  NOT NULL,
    description     VARCHAR(1000) DEFAULT NULL,
    design_json     TEXT          DEFAULT NULL,
    created_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at      TIMESTAMP     NOT NULL DEFAULT CURRENT_TIMESTAMP
);

CREATE INDEX idx_agents_name ON agents(name);

-- 使用触发器实现 updated_at 自动更新
CREATE OR REPLACE FUNCTION update_updated_at_column()
RETURNS TRIGGER AS $$
BEGIN
    NEW.updated_at = CURRENT_TIMESTAMP;
    RETURN NEW;
END;
$$ language 'plpgsql';

CREATE TRIGGER trg_agents_updated_at
    BEFORE UPDATE ON agents
    FOR EACH ROW EXECUTE FUNCTION update_updated_at_column();
```

### 7.3.3 MyBatis-Plus 方言配置

```java
@Configuration
public class MybatisPlusConfig {

    @Bean
    public MybatisPlusInterceptor mybatisPlusInterceptor(
            @Value("${spring.profiles.active:default}") String profile) {

        MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();

        DbType dbType;
        switch (profile) {
            case "dm":
                dbType = DbType.DM;
                break;
            case "kingbase":
                dbType = DbType.KINGBASE_ES;
                break;
            case "gbase":
                dbType = DbType.GBASE;
                break;
            case "highgo":
                dbType = DbType.POSTGRE_SQL; // 瀚高兼容 PostgreSQL
                break;
            default:
                dbType = DbType.MYSQL;
        }

        interceptor.addInnerInterceptor(new PaginationInnerInterceptor(dbType));
        return interceptor;
    }
}
```

### 7.3.4 条件 SQL 编写（Mapper XML）

对于无法通过 ORM 自动适配的 SQL（如 UPSERT），使用 MyBatis `databaseId` 机制：

```xml
<!-- 统计数据 UPSERT — MySQL 版本 -->
<insert id="upsertCallStats" databaseId="mysql">
    INSERT INTO flow_call_stats (flow_code, stat_date, total_calls, success_calls, failed_calls)
    VALUES (#{flowCode}, #{statDate}, #{total}, #{success}, #{failed})
    ON DUPLICATE KEY UPDATE
        total_calls = total_calls + #{total},
        success_calls = success_calls + #{success},
        failed_calls = failed_calls + #{failed}
</insert>

<!-- 统计数据 UPSERT — 达梦 DM8 版本 【定开】 -->
<insert id="upsertCallStats" databaseId="dm">
    MERGE INTO flow_call_stats t
    USING (SELECT #{flowCode} AS flow_code, #{statDate} AS stat_date FROM DUAL) s
    ON (t.flow_code = s.flow_code AND t.stat_date = s.stat_date)
    WHEN MATCHED THEN
        UPDATE SET t.total_calls = t.total_calls + #{total},
                   t.success_calls = t.success_calls + #{success},
                   t.failed_calls = t.failed_calls + #{failed}
    WHEN NOT MATCHED THEN
        INSERT (flow_code, stat_date, total_calls, success_calls, failed_calls)
        VALUES (#{flowCode}, #{statDate}, #{total}, #{success}, #{failed})
</insert>

<!-- 统计数据 UPSERT — 人大金仓 版本 【定开】 -->
<insert id="upsertCallStats" databaseId="kingbase">
    INSERT INTO flow_call_stats (flow_code, stat_date, total_calls, success_calls, failed_calls)
    VALUES (#{flowCode}, #{statDate}, #{total}, #{success}, #{failed})
    ON CONFLICT (flow_code, stat_date) DO UPDATE SET
        total_calls = flow_call_stats.total_calls + #{total},
        success_calls = flow_call_stats.success_calls + #{success},
        failed_calls = flow_call_stats.failed_calls + #{failed}
</insert>
```

## 7.4 缓存中间件适配方案（Redis → 信创缓存）

### 7.4.1 适配范围

| 使用场景 | Redis 命令 | 兼容风险 | 说明 |
|---------|-----------|---------|------|
| 路由缓存 | GET / SET / DEL / EXPIRE | 低 | 基础 KV 操作，信创缓存均兼容 |
| Schema 缓存 | GET / SET / DEL / EXPIRE | 低 | 同上 |
| 分布式锁 | SET NX EX | 低 | 基础原子操作 |
| 调用限流计数 | INCR / EXPIRE | 低 | 基础计数操作 |
| Lua 脚本（限流） | EVAL | **中** | 东方通 TongRDS 对 Lua 支持需逐版本验证 |
| Sentinel 哨兵 | SENTINEL 命令 | **中** | 需确认信创缓存的高可用方案 |
| Cluster 模式 | CLUSTER 命令 | **中** | 需确认信创缓存的集群方案 |

### 7.4.3 高级特性降级方案

当信创缓存不支持某些 Redis 高级特性时，提供降级实现：

| 特性 | 降级方案 |
|------|---------|
| Lua 脚本（限流） | 降级为多次原子命令 + 乐观锁重试；或使用 Guava RateLimiter 单机限流 |
| Redis Stream（消息队列） | 降级为数据库轮询 + 状态表 |
| Sentinel 哨兵 | 使用信创缓存自身的高可用方案（如 TongRDS 主备模式） |
| Cluster 分片 | 使用信创缓存自身的分片方案或单实例 + 应用层分区 |

## 7.5 应用服务器适配方案（Tomcat → 信创中间件）

### 7.5.1 适配说明

Spring Boot 默认内嵌 Tomcat 运行。信创场景需替换为东方通 TongWeb 或中创 InforSuite AS 时，需从内嵌模式改为外置 WAR 部署：

| 改动项 | 内容 |
|--------|------|
| 打包方式 | `pom.xml` 中 `<packaging>` 由 `jar` 改为 `war` |
| 排除内嵌 Tomcat | `spring-boot-starter-web` 中 exclude `spring-boot-starter-tomcat` |
| 启动类适配 | 主类继承 `SpringBootServletInitializer`，重写 `configure()` |
| 部署方式 | 将 WAR 包部署到 TongWeb / InforSuite AS 的应用目录 |
| 配置差异 | 端口、上下文路径、JNDI 数据源等在中间件控制台配置 |

### 7.5.2 WAR 打包改造要点

```java
// 适配外置中间件部署
@SpringBootApplication
public class SmartFlowEngineApplication extends SpringBootServletInitializer {

    @Override

        return builder.sources(SmartFlowEngineApplication.class);

    public static void main(String[] args) {
        SpringApplication.run(SmartFlowEngineApplication.class, args);

```

## 附录 D：错误排查指引

| 现象 | 可能原因 | 排查步骤 |
|------|---------|---------|
| 404 能力不存在 | flowCode 错误或已下架 | 检查 flow_route 表 status 字段 |
| 400 参数校验失败 | 必填参数缺失或类型不匹配 | 查看响应 msg 中的具体字段 |
| 408 执行超时 | 工作流内节点耗时过长 | 查看 task_executions 中 node_logs |
| 502 上游服务异常 | agents-server 不可用 | 检查 agents-server 健康状态 |
| 401 认证失败 | 凭证无效或签名错误 | 检查 SceneCode/AccessKey 有效性、签名算法 |
