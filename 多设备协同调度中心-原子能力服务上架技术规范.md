# 1 引言

## 1.1 编制目的

当前在视联平台的能力建设管理方面，为了规范下游业务原子能力与通用原子服务按照统一的标准、要求、规范完成基础能力服务的上架，做到 **技术统一、流程规范、动作标准**，让多设备协同调度体系的服务能力数量与质量得到进一步改进，特制定本规范。

本规范涵盖：
- 原子能力的版本目录管理规范
- 原子能力的接入标准与技术要求
- 两种标准化上架方式（Knife4j JSON 导入 / Excel 模板导入）
- 导入历史回溯与操作审计机制
- 与能力编排画布的对接规范

## 1.2 适用对象

| 对象 | 职责 |
|------|------|
| **服务提供商** | 负责将各自模块下的原子 API 能力按规范上架至平台 |
| **平台管理员** | 负责管理所有模块下的原子能力，审核上架内容，维护版本目录 |
| **编排人员** | 在场景流程任务中引用已上架的原子能力进行可视化编排 |
| **运营人员** | 查看导入历史、操作日志，进行版本回退等运维操作 |

本标准适用于：

# 4 术语和定义

下列术语和定义适用于本文件。

## 4.1 RESTful

RESTful 是一种网络应用程序的设计风格和开发方式，基于 HTTP，可以使用 XML 格式定义或 JSON 格式定义。

## 4.2 JSON

JSON（JavaScript Object Notation，JS 对象简谱）是一种轻量级的数据交换格式。它基于 ECMAScript（欧洲计算机协会制定的 JS 规范）的一个子集，采用完全独立于编程语言的文本格式来存储和表示数据。简洁和清晰的层次结构使得 JSON 成为理想的数据交换语言。易于人阅读和编写，同时也易于机器解析和生成，并有效地提升网络传输效率。

## 4.3 API

API（Application Programming Interface，应用程序接口）是一些预先定义的接口（如函数、HTTP 接口），或指软件系统不同组成部分衔接的约定。用来提供应用程序与开发人员基于某软件或硬件得以访问的一组例程，而又无需访问源码，或理解内部工作机制的细节。

## 4.4 原子能力

原子能力是指不可再拆分的、最小粒度的基础服务接口。每个原子能力对应一个独立的 REST API 端点，完成一个明确的业务动作（如：查询告警列表、获取设备状态、创建巡查计划等）。原子能力是场景流程编排的最小构建单元。

## 4.5 Knife4j

Knife4j 是基于 Swagger/OpenAPI 的 Java 接口文档增强工具，可自动从 Spring Boot 微服务中生成标准化的接口定义 JSON 文件，包含接口路径、请求方式、请求参数、响应结构等完整信息。

## 4.6 版本目录

版本目录是按地域维度划分的能力版本管理层级。顶级为地域标识（如属地、中心、省级），其下挂载具体的版本号。版本目录是原子能力上架的前提条件。

## 4.7 模块

模块是原子能力的业务分类维度，对应视联平台的功能模块划分，如告警事件能力、GIS 能力、视频能力、设备能力等。模块定义维护在系统数据字典中，与视联平台模块保持一致。

# 5 概述

## 5.1 系统定位

多设备协同调度中心定位为覆盖能力全生命周期的能力引擎平台，覆盖 **能力接入 → 能力编排 → 能力发布 → 能力开放** 完整链路。原子能力服务上架是整个生命周期的起点，是后续编排、发布、开放的"原材料"。

# 6 版本管理规范

## 6.1 版本目录体系

版本目录采用 **地域 → 版本号** 的两级层次结构。地域标识用于区分不同的部署环境或行政区域，版本号管理该地域下的能力集合快照。

```
版本目录树：
├── 属地（Local）
│   ├── 1.8.0（当前版本）
│   ├── 1.8.1（增量更新版本）
│   └── ...
├── 中心（Center）
│   ├── 3.4.0（当前版本）
│   └── ...
├── XX省（Provincial）
│   ├── 1.0.0（引入自中心 3.4.0）
│   └── ...
└── ...
```

| 字段 | 说明 | 约束 |
|------|------|------|
| **地域标识** | 顶级分类，如：属地、中心、省级名称 | 【强制】不可重复，由平台管理员维护 |
| **版本号** | 该地域下的具体版本，遵循 SemVer 规范 | 【强制】格式为 `主版本.次版本.修订号` |
| **引入来源** | 新建版本时可选择引入已有版本的全量能力 | 【可选】引入后可增量修改 |

## 6.2 版本号规范

【强制】版本号遵循语义化版本（Semantic Versioning 2.0.0）：

```
格式：MAJOR.MINOR.PATCH

MAJOR — 主版本号：不兼容的重大变更
MINOR — 次版本号：向后兼容的功能新增
PATCH — 修订号：向后兼容的问题修正
```

示例：
- `1.8.0` → `1.8.1`：修订号变更，小幅修正
- `1.8.1` → `1.9.0`：次版本变更，新增能力
- `1.9.0` → `2.0.0`：主版本变更，不兼容重构

## 6.3 版本新增与继承规则

### 6.3.1 同地域版本新增

当在同一地域目录下新增版本时，系统执行以下操作：

```
操作：属地下新增版本 1.8.1

┌──────────┐         ┌──────────┐
│ 属地     │         │ 属地      │
│ 1.8.0   │  ────→  │ 1.8.0    │
│ (85个能力)│         │ (85个能力) │
└──────────┘         │ 1.8.1    │
                     │ (85个能力) │  ← 自动从 1.8.0 全量 Copy
                     └──────────┘
```

| 规则 | 说明 |
|------|------|
| **自动继承** | 【强制】新版本自动继承前一版本的全量能力接口（含请求头、入参、出参、示例等完整文档） |
| **继承来源** | 【强制】默认继承该地域下最新版本（即当前版本）的全部能力 |
| **增量更新** | 【强制】继承后通过模板导入或 Knife4j 导入进行增量更新 |
| **独立快照** | 【强制】新版本与源版本数据独立，修改不互相影响 |

### 6.3.2 跨地域版本引入

当新增其他地域版本目录后创建首个版本时，可选择引入已有地域版本的全量能力：

```
操作：新建 XX省 版本目录，新增版本 1.0.0，引入中心 3.4.0

┌──────────┐                     ┌──────────┐
│ 中心     │     引入             │ XX省     │
│ 3.4.0   │  ──────────────→    │ 1.0.0    │
│ (N个能力) │                     │ (N个能力) │  ← 具备 3.4.0 全部能力
└──────────┘                     └──────────┘
```

| 规则 | 说明 |
|------|------|
| **引入选择** | 【可选】新版本可选择引入任意地域下任意版本的全量能力 |
| **引入记录** | 【强制】系统记录引入来源（源地域 + 源版本号），便于溯源 |
| **独立演化** | 【强制】引入后两个版本各自独立演化，不自动同步 |

## 6.4 版本引入机制

版本引入的完整流程：

```
1. 平台管理员 在版本管理页面 新建地域版本目录（如：XX省）
2. 在该目录下 新增版本号（如：1.0.0）
3. 弹出引入版本选择器：
   ┌─────────────────────────────────────┐
   │  引入版本（可选）                      │
   │                                     │
   │  □ 不引入，创建空白版本                │
   │  ● 引入已有版本                       │
   │                                     │
   │  地域：[ 中心       ▼ ]              │
   │  版本：[ 3.4.0      ▼ ]              │
   │                                     │
   │  将引入 577 个原子能力接口              │
   │                                     │
   │       [ 取消 ]    [ 确认创建 ]         │
   └─────────────────────────────────────┘
4. 确认后系统将源版本全量能力数据 Copy 至新版本下
5. 后续通过标准导入流程进行增量更新
```

## 6.5 版本状态流转

```
                 创建              激活              归档
    (Draft) ──────────→ (Active) ──────────→ (Archived)
      草稿                 生效                 已归档
       │                   │
       │    删除            │    废弃
       ▼                   ▼
    (Deleted)          (Deprecated)
      已删除               已废弃
```

| 状态 | 说明 | 允许操作 |
|------|------|---------|
| **Draft** | 草稿，新建后默认状态 | 编辑、导入、激活、删除 |
| **Active** | 生效，当前使用版本 | 导入更新、归档、废弃 |
| **Archived** | 归档，历史保留 | 仅查看 |
| **Deprecated** | 废弃，不再使用 | 仅查看 |
| **Deleted** | 已删除 | 不可见 |

【强制】同一地域下同一时间仅允许一个 Active 状态的版本。

| **编排人员** | 只读引用已上架能力 | 在编排画布中浏览、搜索、拖拽原子能力 |

### 服务提供商登录视角

服务提供商以其身份登录后，系统根据其关联的微服务名（如 `video-video`）自动过滤展示其负责模块下的能力列表。服务提供商只能导入和维护其权限范围内的模块能力。

### 平台管理员登录视角

平台管理员登录后可看到全量能力视图，按模块分组展示所有原子能力，并可进行跨模块的统一管理操作。

## 7.2 业务研发技术规范

业务系统在建设时推荐使用微服务架构，使用 REST 接口完成微服务间调用。

| 级别 | 规范项 |
|------|--------|
| 【推荐】 | 采用微服务架构风格进行开发 |
| 【强制】 | 对接平台提供的运维监控中心、性能日志中心 |
| 【强制】 | 开放服务统一使用 REST 接口的形式提供，并提供满足 OpenAPI 规范的接口文档 |
| 【强制】 | 微服务需在 Nacos 注册中心完成服务注册，服务名遵循 `video-{模块名}` 命名规范 |
| 【强制】 | 接口定义遵循本规范中的请求方式、传参方式、返回格式等要求 |
| 【推荐】 | 集成 Knife4j（Swagger 增强）自动生成接口文档 JSON |

## 7.3 HTTP 请求方式

接口分成读、写两种类型，分别使用 GET、POST 两种请求方式：

| 方法 | 用途 | 说明 |
|------|------|------|
| **GET** | 读操作 | 从服务器取出资源（一项或多项），查询类接口 |
| **POST** | 写操作 | 更改服务器资源——新增、修改或删除 |
| **PUT** | 全量更新 | 更新资源的全部字段（可选使用） |
| **DELETE** | 删除 | 删除指定资源（可选使用） |

## 7.4 API 过滤参数

API 应该提供参数过滤返回结果，使接口返回的结果可控可预期。URL 后由 `?参数名=参数值` 构成，参数示例如下：

| 参数示例 | 说明 |
|---------|------|
| `?limit=10` | 指定返回记录的数量 |
| `?offset=10` | 指定返回记录的开始位置 |
| `?page=2&rows=100` | 指定第几页，以及每页的记录数 |
| `?sortby=name&order=asc` | 指定返回结果按照哪个属性排序，以及排序顺序 |
| `?permission_type=1` | 指定筛选条件 |

## 7.5 API 传入参数

提供 3 种不同的传参方式：

| 方式 | 位置 | 适用场景 | 说明 |
|------|------|---------|------|
| **Header** | 请求头 | 鉴权、会话、元数据 | Content-Type、Authorization 等 |
| **Param（Query）** | URL 查询参数 | 简单过滤、分页 | GET 请求的查询条件 |
| **Body** | 请求体 | 复杂业务参数 | POST 请求的 JSON 对象 |

【强制】服务提供商可根据自身业务实现需要选择合适的方式，在服务能力上线后传参方式不可改变。

## 7.6 返回数据格式

为保障前后端数据交互的顺畅，**【强制】** 采用以下固定的数据格式封装：

```json
{
    "code": 200,
    "isError": false,
    "message": "操作成功",
    "data": {}
}
```

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `code` | Integer | 是 | 接口执行的状态码，接口需定义状态码并在接口文档中体现 |
| `isError` | Boolean | 是 | 接口执行失败标识，作为成功或失败的直接判断 |
| `message` | String | 是 | 执行失败时返回的错误信息；成功时可返回 "操作成功" |
| `data` | Object/Array | 否 | 接口返回的主数据，可根据实际情况返回数组或 JSON 对象 |

## 7.7 错误码规范

【强制】原子能力接口需遵循以下统一错误码体系：

| code 范围 | 说明 | 示例 |
|-----------|------|------|
| 200 | 成功 | 操作成功 |
| 400~499 | 客户端错误 | 参数校验失败(400)、未授权(401)、无权限(403)、资源不存在(404) |
| 500~599 | 服务端错误 | 系统内部错误(500)、上游服务异常(502)、服务不可用(503) |

【推荐】各模块可在上述基础上扩展业务错误码，格式建议为 `模块前缀 + 4位数字`，如 `ALARM_1001`。

### 8.2.2 导入流程

┌──────────────────────────────────────────────────────────────────────┐
│  Knife4j JSON 导入流程                                                │

│  2. 进入能力上架页面 → 选择目标版本（地域 + 版本号）                       │

| `paths[].description` | 能力描述 | 直接取值 |

## 8.4 API 能力上架 Excel 模板定义

### 8.4.1 设计原则

模板遵循以下原则：

| 原则 | 说明 |
|------|------|
| **模块级默认** | 请求头、鉴权方式、Content-Type、通用错误码等在模块级统一配置，单接口不再重复填写 |
| **一行一接口** | 主表一行代表一个原子能力，核心字段控制在 10 列以内 |
| **参数紧凑表达** | 入参/出参采用单元格内紧凑 DSL 语法描述，无需独立 Sheet |
| **系统自动补全** | 请求示例、响应骨架、通用请求头由系统根据参数定义自动生成 |
| **只填差异项** | 符合默认约定的字段留空即可，系统按约定填充 |

### 8.4.2 Sheet 1：能力清单

每行一个原子接口，核心信息一目了然：

| 列号 | 列名 | 必填 | 说明 | 填写示例 |
|------|------|------|------|---------|
| A | 能力编码 | **是** | 唯一编码。留空则系统按 `路径+方法` 自动生成 | `alarm_query_list` |
| B | 能力名称 | **是** | 中文可读名称 | 告警列表查询 |
| C | 所属模块 | **是** | 匹配数据字典中的模块名（数据下拉） | 告警事件能力 |
| D | 请求方式 | **是** | GET / POST | POST |
| E | 接口路径 | **是** | URI 路径（不含域名） | `/api/alarm/list` |
| F | 能力描述 | **是** | 一句话说明接口功能 | 根据条件分页查询告警列表 |
| G | 入参简述 | **是** | 紧凑语法（见下方说明），留空则仅导入接口骨架，参数后续在页面维护 | `alarmTimeStart:String:必填, pageNum:Integer:必填=1, pageSize:Integer:必填=10` |
| H | 出参简述 | **是** | data 域内的业务字段简述，通用外壳（code/message）无需填写 | `total:Integer, list[].processId:String, list[].cityName:String` |
| I | 微服务名 | 否 | 留空则按模块映射表自动匹配 | `video-alarm` |
| J | 备注 | 否 | 补充说明 | |

**系统自动补全规则**（以下字段无需手工填写）：

| 自动补全项 | 补全逻辑 |
|-----------|---------|
| Content-Type | 默认 `application/json`，GET 请求无 Body 时不设置 |
| 鉴权方式 | 继承模块级配置（默认 TOKEN） |
| 请求头 | 继承模块级通用请求头模板（Content-Type + Authorization） |
| 通用错误码 | 继承平台统一错误码集（200/400/401/403/404/500） |
| 超时时间 | 默认 30 秒 |
| 请求示例 | 根据入参自动生成 JSON 骨架 |
| 响应示例 | 根据出参自动生成 `{"code":200,"data":{...}}` 骨架 |
| 微服务名 | 根据模块-微服务映射表自动匹配 |
| 能力编码 | 留空时按 `路径转下划线_方法` 自动生成 |

### 8.4.3 入参/出参紧凑语法

为避免为每个参数单开一行，采用单元格内紧凑 DSL 语法，逗号分隔多个参数：

```
语法格式：参数名:类型[:必填][=默认值][ 说明]

类型简写：
  String / Integer / Long / Double / Boolean / Object / Array
  不写类型默认 String

必填标识：
  :必填   → required = true
  不写    → required = false

默认值：
  =值    → 设置默认值

说明：
  空格后跟说明文字（可选）

嵌套写法：
  对象：   obj.fieldName:Type
  数组元素：list[].fieldName:Type
```

**入参示例**（G 列）：

```
alarmTimeStart:String:必填 告警开始时间,
alarmTimeEnd:String:必填 告警结束时间,
pageNum:Integer:必填=1 页码,
pageSize:Integer:必填=10 每页条数,
alarmLevel:String 告警级别
```

**出参示例**（H 列，只写 data 域内的业务字段）：

```
total:Integer 总记录数,
list[].processId:String 流程ID,
list[].cityName:String 城市名称,
list[].latitude:String 纬度,
list[].orderStatus:String 工单状态
```

> 系统解析后自动包裹统一响应外壳 `{"code":200,"isError":false,"message":"...","data":{这里}}`，无需重复定义 code/message 等通用字段。

### 8.4.4 模块级默认配置

以下配置在平台管理端的 **模块配置页面** 维护，所有属于该模块的接口自动继承，无需在 Excel 中重复填写：

| 配置项 | 说明 | 默认值 |
|--------|------|--------|
| 通用请求头 | 该模块所有接口共用的 Header | `Content-Type: application/json`、`Authorization: Bearer {token}` |
| 鉴权方式 | TOKEN / ACCESS_KEY / NONE | TOKEN |
| Content-Type | 请求内容类型 | `application/json` |
| 超时时间 | 默认超时（秒） | 30 |
| 通用错误码 | 继承平台统一错误码 + 模块扩展错误码 | 200/400/401/403/404/500 |

# 11 服务能力发布与审核规范

## 11.1 能力状态生命周期

原子能力从导入到最终开放使用，经历完整的状态生命周期管理：

未上架(DRAFT) → 已上架(SHELVED) → 已发布(PUBLISHED) → 已下架(OFFLINE)
                                       ↑                    │
                                       └── 重新发布 ─────────┘

| 状态 | 编码 | 说明 | 允许操作 |
|------|------|------|---------|
| **未上架** | DRAFT | 初始导入但未提交的能力 | 编辑、删除、提交上架 |
| **已上架** | SHELVED | 已完成信息维护，可提交审核 | 编辑、提交审核、删除 |
| **已发布** | PUBLISHED | 审核通过，对编排和市场可见 | 下架、废弃 |
| **已下架** | OFFLINE | 临时下架，不对外可见 | 重新上架、编辑 |

## 11.3 能力下架与废弃

### 下架规则

| 规则 | 说明 |
|------|------|
| **下架前置检查** | 系统自动检测该能力是否被已发布的编排流程引用，若有引用须二次确认 |
| **优雅下架** | 下架后已引用该能力的编排流程不受影响（保持快照），但新编排不可再引用 |
| **下架通知** | 下架操作自动通知所有引用方（通过消息中心/邮件） |
| **重新上架** | 已下架的能力可重新上架，无需重新审核（30天内）；超期需重新审核 |

### 废弃规则

| 规则 | 说明 |
|------|------|
| **废弃条件** | 仅平台管理员可执行废弃操作 |
| **不可逆** | 废弃操作不可逆，数据保留但状态不可变更 |
| **废弃通知** | 自动通知所有相关方，建议迁移至替代能力 |
| **替代指引** | 废弃时须填写替代能力编码（如有） |

| **结果展示** | 展示完整的请求头、请求体、响应头、响应体、HTTP 状态码、耗时 |
| **历史记录** | 保存最近 50 次调试记录，可快速重放 |
| **cURL 导出** | 一键复制 cURL 命令 |

### 12.4.2 调试安全约束

| 约束项 | 说明 |
|--------|------|
| 登录要求 | 【强制】必须登录后才能使用调试功能 |
| 频率限制 | 【强制】调试接口限流：单用户 10 次/分钟 |
| 数据隔离 | 【强制】调试请求走独立的调试通道，与生产流量隔离 |
| 敏感脱敏 | 【强制】调试结果中的敏感字段自动脱敏 |

## 12.5 安全与鉴权规范

### 12.5.1 鉴权方式

| 鉴权方式 | 适用场景 | 说明 |
|---------|---------|------|
| **Token 认证** | 视联平台内部调用 | 通过统一认证中心签发的 Bearer Token |
| **AccessKey 签名** | 第三方系统/能力开放平台调用 | HMAC-SHA256 签名认证 |
| **无鉴权** | 微服务集群内部调用 | 仅限可信内网环境 |
